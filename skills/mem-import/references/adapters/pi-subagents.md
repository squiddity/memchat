# Known adapter profile: pi-subagents

Load this reference only when the active tool catalog or installed local documentation identifies `pi-subagents`.

It is a possible account-level host-agent facility when the user has installed and enabled it. Its documented surface includes fresh/forked workers, async/foreground modes, parallel/chain/fanout forms, timeouts/budgets, run IDs, lifecycle artifacts, telemetry, waiting, strict profile allowlists, and package-distributed worker profiles.

## U0 use

- Inspect the installed version and actual tool schema; do not assume the planned version or every documented field is available.
- Choose a single fresh, low-risk scout task when its native controls make that reasonable.
- Use the extension's own task/run identity and wait/result mechanism when available.
- Request the narrowest available profile/tool configuration, but inspect the selected builtin profile rather than assuming its name implies a read-only allowlist. Record the request as unproved unless observed enforcement is later tested.
- For a non-mutating scout, do not use an acceptance setting that requires changed-file or mutation evidence; use no acceptance policy or one that accepts a read-only receipt.
- Give the scout sufficient bounded turns to read every named file and return once. If the budget prevents that, diagnose the configuration rather than treating the result as a facility verdict.
- Record version, requested context/profile, correlation behavior, completion behavior, and limitations for the parent decision.

## U1 extraction use

Use it only after the parent has confirmed that the spawned child can see the `mem-import` typed-tool extension in the active host/project resource configuration. The parent supplies the extractor role prompt and assignment bootstrap, requests a fresh context and the narrowest profile/tool configuration available, then waits through the adapter's native mechanism.

The U0 compatibility trial showed that builtin profiles may expose more tools than a role name suggests and that an acceptance policy can reject a harmless read-only result. Inspect the actual installed profile/settings, avoid mutation-oriented acceptance rules for read-only work, and record limits rather than assuming enforcement.

A raw assignment grant may need to travel in the worker task/bootstrap because the generic Pi tool API has no transcript-secret channel. Keep it out of durable world artifacts and receipts; treat this as an application authorization boundary, not a credential vault.

## Not part of U1

Do not pin/install this dependency under memchat, create a CLI host, choose a global worker backend, or claim subprocess hermeticity. Those remain host-policy decisions driven by observed outcomes.
