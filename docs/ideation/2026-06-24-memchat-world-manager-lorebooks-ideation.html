Ideation: memchat world manager & lorebooks

Ranked directions for turning memchat from a memory backend experiment into a traceable world manager that can ingest source material, maintain canon, surface conflicts, and assemble the right lore for long-running chats.

Grounding Context

Codebase context. Memchat already has the right conceptual substrate for a world manager: append-only transcript audit logs, qmd-compatible markdown memory, explicit fact/state/conflict separation, current-session precedence, grouped conflict rendering, ignore/tombstone behavior, and flush/compaction boundaries. The repo is still small enough that a world-memory layer can become a first-class domain abstraction instead of disappearing into more conditionals inside memory.ts.

External context. Prior art converges on layered memory, not one universal store. SillyTavern shows the value of scoped lorebooks, activation rules, recursion, and budget-aware prompt injection. Cloudflare Sessions, MongoDB/LangGraph, Redis, and AdaMem all reinforce the split between raw history, compact working memory, searchable episodic memory, and curated long-term knowledge. The opportunity gap is clear: lorebook systems are weak on provenance and canon governance, while generic agent-memory stacks are weak on narrative canon, reveal timing, and retcons.

Qualification standard. Surviving ideas were favored when they built directly on memchat’s existing traceability goals, had a concrete meeting-test, and improved the product’s identity as a world manager rather than just a better recall backend.

Provenance-Preserving World Library

#1 Source material ingestion and archive creation Confidence 82% Complexity High

Description: Treat memchat as both a searchable library of full source artifacts and a curated canon editor. Imported setting guides, transcripts, campaign notes, and reference docs remain loadable whole, while the system also derives smaller lore packets, state candidates, and source-linked canon snippets from them.

Basis: external: Strong prior art supports keeping whole source artifacts distinct from distilled reusable memory, and memchat already values exact-source fallback and provenance-preserving separation.

Rationale: This is the biggest product-identity shift in the set. It gives memchat a durable answer to “what is the world made of?” and makes exact-source verification a feature, not a recovery path.

Downsides: Requires import pipelines, source indexing, and a disciplined distinction between archival artifacts and prompt-ready canon; otherwise it becomes an expensive filing cabinet.

Conflict / Retcon Ledger with Addenda and Auto-Closure

#2 Update governance, conflicts, and retcons Confidence 89% Complexity Medium

Description: Promote contradictions into first-class ledger entries with competing claims, timestamps, sources, statuses, and amendment history. Straightforward cases can auto-close when newer evidence converges; ambiguous ones remain visible until reviewed.

Basis: direct: The repo already detects retcon cues, groups conflicts separately in injected context, preserves provenance, and explicitly points toward durable conflict events as a next step.

Rationale: This is the strongest direct extension of memchat’s current behavior. It upgrades “possible conflicts or retcons” from prompt-time presentation into durable world governance infrastructure.

Downsides: Over-triggering would create noise; it needs clear resolution states and a lightweight operator workflow to avoid turning every inconsistency into paperwork.

Worldboard + Codex Layered Memory

#3 Lore/state data model and storage layers Confidence 84% Complexity Medium

Description: Split durable memory into a compact mutable Worldboard for “what is true right now” and a broader Codex for stable background lore. Raw transcripts and imported sources remain separate provenance layers underneath both.

Basis: direct: Memchat already separates stable facts from current state; formalizing that into two first-class layers is a grounded next abstraction.

Rationale: This makes the archive legible to both humans and agents. It also reduces retrieval noise by making live continuity and deep lore different things on purpose.

Downsides: Boundary mistakes will confuse both users and retrieval logic; the system needs disciplined definitions for when something belongs in the board, the codex, or both.

Context Compiler with Dry-Run Budget Trace

#4 Retrieval and prompt-context injection Confidence 80% Complexity High

Description: Replace opaque “top hits” retrieval with a compiler that assembles prompt context from explicit lanes such as current state, stable lore, branch deltas, conflict packets, and exact-source evidence, while exposing a dry-run trace of why each block activated and what it cost.

Basis: direct: Retrieval is already multi-stage and debug-visible in memchat, so a compiler-like assembly model extends existing mechanics rather than inventing a new subsystem from nowhere.

Rationale: If world memory gets richer, trust depends on inspectable assembly. This idea turns retrieval tuning from folklore into something observable and improvable.

Downsides: More instrumentation and routing logic means more surface area to maintain; poor trace design could overwhelm users instead of helping them.

Non-Canon Ledger with Auto-Quarantine

#5 Update governance, conflicts, and retcons Confidence 86% Complexity Medium

Description: Preserve mistaken turns, rejected extractions, abandoned canon, and obvious accidental input in a first-class anti-canon layer that retrieval and synthesis consult to avoid resurrecting them. Mistake handling becomes durable memory governance instead of a one-off cleanup command.

Basis: direct: Ignore annotations, retrieval filtering, and mistake heuristics already exist, so formal anti-canon memory is directly supported by repo behavior.

Rationale: Durable memory needs durable rejection, not just durable acceptance. This idea turns bad claims into reusable guardrails.

Downsides: Too-aggressive quarantine could hide useful but awkward information; restore and audit flows need to stay simple.

Live Scene Compiler with Scope and Reveal Control

#6 Retrieval and prompt-context injection Confidence 74% Complexity High

Description: Build prompt context as a scene-specific packet that respects scope, POV, and reveal state. Canon can be true in the archive without always being injectable into the current scene.

Basis: external: Scoped lorebooks, reveal gates, and layered prompt assembly are proven elsewhere and map cleanly onto memchat’s state-vs-lore split.

Rationale: This is the strongest narrative-facing idea in the set. It solves the real storytelling problem: not just knowing the truth, but deciding which truth belongs in the current moment.

Downsides: Scope and reveal rules can become complex quickly; bad gating logic would create invisible bugs where the system “knows” something but surfaces the wrong slice.

World Operations Workbench and /world Command Surface

#7 Agent workflows, commands, and background processing Confidence 68% Complexity Medium

Description: Expose the world-manager layer through a first-class operational surface — commands such as /world import, /world inbox, /world merge, /world conflicts, and /world inspect — plus a workbench view for queue health, pending patches, and source drilldown.

Basis: reasoned: Memchat already has /memory commands, work-status data, and debug surfaces; a world-manager product needs an operator-facing surface built on that substrate.

Rationale: A world manager that only exists as hidden plumbing will be hard to trust, debug, and iterate. This idea makes the archive feel like a product capability instead of internal machinery.

Downsides: It adds user-facing surface area before the deeper data model is settled, so the command design must stay simple enough to survive iteration.
